專業指南：利用 Swift Package Manager (SPM) 實現 iOS 應用程式模組化架構

1. 導論：規模化開發下的瓶頸與編譯防火牆

在維護每月活躍用戶超過 550,000 名的大型 iOS 專案時，單體式架構（Monolithic Architecture）會不可避免地成為研發效率的枷鎖。當代碼庫膨脹到一定規模，最直觀的損害在於開發者體驗（Developer Experience）的崩潰：Xcode Previews 因為全量編譯的負擔而頻繁失效，甚至完全無法運作。這不僅是等待時間的問題，而是阻斷了 UI 迭代的即時回饋循環。

作為系統架構師，我們必須建立「編譯防火牆（Compilation Firewall）」。透過模組化（Modularization），我們將龐大的專案拆解為具備明確邊界的 Swift 套件，其核心價值在於：

* 戰略級代碼複用 (Code Reuse)： 基礎設施如驗證邏輯（Authentication）、網路層（Networking）與設計系統（Design System）不應只存在於主 App 中。模組化讓這些組件能無縫共享至 Widget、Watch App 甚至其他獨立產品線，消滅重複造輪子的成本。
* 聚焦編譯與開發速度 (Focus Compilation)： 透過獨立套件，開發者可以在類似「微型 App（Micro-App）」的環境下工作。當你在開發單一 Feature 模組時，Xcode 僅需編譯該模組及其依賴項，徹底解決 Previews 載入過慢的痛點，實現真正的快速迭代。

明確的架構分類是成功實作模組化的前提，否則只是將混亂從 Project 轉移到 Package。

2. 核心部署策略：本地 SPM 套件的戰略優勢

在實作路徑上，我強烈建議採用「本地 (Local) SPM 套件」而非遠端倉庫。對於頻繁迭代的企業級專案，本地套件提供了開發流暢度與版本控制之間的完美平衡。

決策矩陣：本地 vs. 遠端 SPM 套件

維度    本地 SPM 套件 (Local)    遠端 SPM 套件 (Global)
更新成本    極低：隨改隨用，與主專案同步編譯    高：需提交代碼、標記 Tag、更新依賴版本
整合效率    即時整合，無需網路解析    依賴網路狀態，解析時間較長
架構意圖    專屬於該專案的業務邏輯、功能模組    通用型工具庫、跨產品線共享、開源組件
開發體驗    支援「聚焦編譯」，有利於快速迭代    適合穩定、不常更動的核心底層

本地套件讓我們在維持單一倉庫（Monorepo）管理便利性的同時，享受模組化帶來的依賴解耦與編譯加速。

3. 三層架構體系：確保低耦合的「規則參與」

結構化的意義在於建立嚴格的依賴階層，防止代碼演變成糾纏不清的義大利麵。我們將模組分為以下三個層級：

層級定義與技術規範

1. Base (基礎層)：
  * 定位： 零依賴或僅依賴外部原子級工具。
  * 範例： MovieModels（純數據結構）、AppConstants（顏色、間距定義）、PreviewHelpers。
  * 價值： 保持極度輕量，確保 UI 邏輯或網路請求不會滲透進最底層的資料定義中。
2. Infrastructure (工具層)：
  * 定位： 依賴 Base，提供技術支撐的組件。
  * 範例： Networking（負責資料獲取）、Navigation（定義跳轉合約）、DesignSystemComponents。
  * 關鍵決策： 雖然 Networking 需要 Models，但反過來需要 Models 的其他模組不應被強制依賴 Networking 實作。
3. Features (功能層)：
  * 定位： 獨立的業務單元，依賴上述兩層。
  * 範例： MovieDetails（詳情頁面）、UserSetting。

依賴禁令 (Rules of Engagement)

* 禁止向上依賴： Base 嚴禁依賴 Infrastructure 或 Features。
* 禁止跨層依賴： Infrastructure 嚴禁依賴 Features。
* 絕對禁止循環依賴 (Circular Dependencies)： 如果套件 A 與 B 互相依賴，代表它們的邊界劃分失敗，應考慮合併或提取共有邏輯至 Base。

4. 實作流程指南：從提取到權限封裝的 SOP

將代碼提取至套件不僅是檔案搬移，更是一次嚴格的 API 表面積（API Surface Area）審查過程。

3. 專家提示 (Pro-tip)： 對於外部依賴，如用於類型安全 ID 的 Tag 程式庫，應在 Base 模組（如 MovieModels）中宣告，而非由各 Feature 自行引入。
4. 存取權限的重新定義： 移入套件後，原有的 internal 屬性將無法被主 App 存取。你必須顯式標註 public。
  * 警告： public 應該是一個自覺的架構決定，而非修復編譯錯誤的反射動作。僅暴露必要的介面，以維持良好的封裝性與減少命名衝突。
5. 驗證編譯隔離： 切換 Scheme 至該套件進行獨立編譯。如果能在此模式下通過編譯，代表該模組已具備「微型 App」的開發條件。

5. 進階技術解析：依賴反轉與導航解耦

模組化最棘手的挑戰在於「功能模組間互不相識，卻需互相跳轉」。這需要運用依賴反轉 (Dependency Inversion) 的原則。

導航套件 (Navigation Package) 的運作邏輯

我們建立一個獨立的 Navigation 套件，它只包含「抽象定義」，而不包含「實體畫面」：

1. 定義目的地 (Destinations)： 使用 Enum 宣告所有可能的跳轉目標（如 case movieDetails(id: Movie.ID)）。
2. 移除 UI 依賴： 在提取過程中，若 Enum 的關聯值包含 UI 類型（如 ImageContainerViewData），必須進行重構。將其改為依賴基礎層的 Model 類型。這樣可以確保 Navigation 套件保持純粹，不被複雜的 UI 組件綁架。
3. App 作為組合器 (Composer)： 具體的 View 映射邏輯（即 Destination 轉變為真實 View 的過程）必須保留在 主 App Target 中。因為只有主 App 擁有全域上下文，知道所有功能模組的存在，它負責完成最後的拼圖。

這種模式允許 Feature A 在完全不知道 Feature B 實作細節的情況下，透過 Navigation 套件發出跳轉請求，實現真正的水平解耦。

6. 結論：邁向可持續發展的架構轉型

實施 SPM 模組化雖然在初期會面臨存取控制權限（Public/Internal）的重構挑戰，但這是一次對代碼品質的深度清理。

核心收益檢查清單 (Checklist)

* [ ] 開發效能： 透過「聚焦編譯」讓 Xcode Previews 恢復秒級回饋。
* [ ] 結構品質： 強迫實施單向依賴，消滅模組間的隱性耦合。
* [ ] 封裝安全性： 縮小 API 暴露表面積，降低大型團隊協作時的代碼污染。
* [ ] 資源獨立化： 每個套件可擁有專屬的 Localization、Asset 與單元測試，便於維護。

最後建議： 不要試圖一次性將整個 App 模組化。從最明顯的 Base（如 Constants 與 Models）開始，逐步提取 Infrastructure，最後再處理 Feature 之間的解耦。這是一場馬拉松，而非百米衝刺。
